# AD001 — Mini HAL architecture: layered pure core, IDF-under-Arduino driver, and configurable deep-sleep power domains

- **Status:** Accepted
- **Date:** 2026-07-03
- **Deciders:** Jesus Carrillo
- **Tags:** architecture, low-power, testability, esp32

## Context

`compute-board` is a KiCad baseboard built around an **ESP32-WROOM-32E**. Every peripheral
and add-on board is powered from a **secondary rail `VCC_AUX`**, which is enabled by an
active-high `XC6220` LDO. The LDO enable is driven by **GPIO13 (`VCC_AUX_ENA`)** and has a
board-level pull-up, so the rail is **ON by default at boot**. A momentary config button
(`SW2`) is wired to **GPIO12 (`CONFIG_ENA`)** with a pull-up and RC debounce, giving an
**active-low** signal. GPIO12 is also the MTDI flash-voltage strapping pin.

The pin map used throughout this library was traced directly from
`compute-board.kicad_sch` (net-to-pin connectivity), not guessed:

| Function | GPIO | Notes |
|---|---|---|
| `VCC_AUX_ENA` (secondary rail enable) | 13 | Active-high, board pull-up → rail ON by default |
| `CONFIG_ENA` (config button) | 12 | Active-low; also MTDI strapping pin |
| LED1 | 15 | On-board LED (driven on/off; see PINOUT) |
| I²C | SDA=21, SCL=22 | |
| SPI / microSD | CS=5, SCK=18, MISO=19, MOSI=23 | |
| UART0 | RXD0=3, TXD0=1 | |
| Broken-out header GPIOs | 2,4,16,17,25,26,27,32,33,34,35,36,39 | 34–39 are input-only |

The sibling project `lora-sensor` already proves a working ultra-low-power deep-sleep
recipe on the same hardware family (`lora-sensor/src/main.cpp`):

1. Put every peripheral I/O into **Hi-Z** (`gpio_set_direction(INPUT)` + float + disable
   pulls) and **latch it through sleep** with `gpio_hold_en()`.
2. `gpio_reset_pin()` the always-on / strapping pins.
3. Turn off RTC power domains via `esp_sleep_pd_config(... OFF)`.
4. Drive the rail-enable **LOW**, `gpio_hold_en()` it, then `gpio_deep_sleep_hold_en()`
   so the rail stays collapsed for the entire sleep.

We want to package this into a reusable **PlatformIO / Arduino library** that is genuinely
unit-testable. Three hard constraints shape the design:

- **UNBREAKABLE #1** — It must be a PlatformIO library for the Arduino SDK, and should also
  be usable from the Espressif IDF SDK.
- **UNBREAKABLE #2** — Hardware must **not** be mocked. We must be able to unit-test most of
  the library without fabricating fake `gpio_*` implementations.
- **UNBREAKABLE #3** — Proper, idiomatic C and C++.

## Decision

### 1. Two-layer split: a pure host-testable *core* and a thin ESP32-only *driver*

The library is divided so that all decision-making logic is free of hardware headers:

- **Pure core** (`src/board_pins.h`, `src/power_plan.{h,cpp}`) — the pin map, the default
  pin sets, the deep-sleep *plan* builder (`SleepPlan`), the config-probe decode
  (`decodeConfigLevel`), and the RTC power-domain configuration type. These translation
  units include **no** `<Arduino.h>` and **no** `driver/gpio.h`; they compile and run on a
  desktop host.
- **Thin driver** (`src/ComputeBoardHal.{h,cpp}`) — the only code that touches hardware.
  It asks the core *what* to do and then performs the corresponding `gpio_*` /
  `esp_sleep_*` calls. It is compiled only for the ESP32 target (guarded with
  `#if defined(ESP32)`).

Because the core computes *what* should happen (which pins go Hi-Z, in what order, which
level the rail ends at, whether the button is pressed) as plain data, we can assert on that
data on the host. There is nothing to mock: the tests exercise the **real production code**,
just not the four-line hardware tail that actually pokes the registers.

### 2. ESP-IDF APIs for sleep/hold; Arduino digital API for level I/O, with an IDF fallback

The Hi-Z-latch and hold-through-sleep behaviour has no Arduino equivalent, so the deep-sleep
path uses ESP-IDF directly: `gpio_hold_en`, `gpio_deep_sleep_hold_en`,
`gpio_set_direction(INPUT)`, `esp_sleep_pd_config`, `esp_deep_sleep_start`.

For plain level I/O — driving `VCC_AUX_ENA` (GPIO13) high/low and reading `CONFIG_ENA`
(GPIO12) — the driver uses the Arduino digital API (`pinMode`/`digitalWrite`/`digitalRead`)
when building under Arduino, with a bare-IDF fallback (`gpio_reset_pin` +
`gpio_set_direction`/`gpio_set_level` / `gpio_get_level`) behind `#if defined(ARDUINO)`.
This was a **deliberate correction**: GPIO13 (and GPIO12) are RTC-capable pads, and bare
`gpio_set_direction`/`gpio_set_level` does not reliably take such a pad high — the rail
stayed unpowered on real hardware. Arduino's `pinMode` performs the extra pad-mux/reset work
that the RTC pad needs, and it matches the proven sequence in the sibling `lora-sensor`
firmware (which drives the same GPIO13 rail enable). Dual-SDK support is preserved by the
`#else` IDF fallback, so the source still compiles under `framework = espidf`.

### 3. Configurable RTC power domains, default all-OFF, expressed in the pure layer

`esp_sleep_pd_config()` takes IDF enums (`esp_sleep_pd_domain_t`, `esp_sleep_pd_option_t`).
To keep those types out of the host-testable header, the pure layer defines a HAL-local
mirror and an aggregate config with **defaults of `Off` for every domain** — byte-for-byte
the current lora-sensor maximum-savings behaviour:

```cpp
enum class PowerDomainOption : std::uint8_t { Off, On, Auto };   // mirrors ESP_PD_OPTION_*

struct RtcPowerConfig {                     // aggregate: zero-cost, brace-initialisable
    PowerDomainOption rtc_periph   = PowerDomainOption::Off;
    PowerDomainOption rtc_slow_mem = PowerDomainOption::Off;
    PowerDomainOption rtc_fast_mem = PowerDomainOption::Off;
    PowerDomainOption xtal         = PowerDomainOption::Off;

    static constexpr RtcPowerConfig maxSavings();     // all Off (the default)
    static constexpr RtcPowerConfig keepRtcMemory();  // retain RTC slow/fast memory
};
```

Callers override it either per call
(`deepSleepSeconds(s, RtcPowerConfig::keepRtcMemory())`) or once via
`setRtcPowerConfig(cfg)`. The driver owns the **only** IDF coupling: a small `constexpr`
table mapping each struct field to its `{esp_sleep_pd_domain_t, esp_sleep_pd_option_t}`
pair. This is what lets an application retain RTC memory (for RTC-retained variables) or
keep a domain powered for a wake stub, without giving up the safe default for the common
case.

### 4. Deep-sleep contract (invariants the core guarantees)

- All registered peripheral pins plus the known on-board bus pins (I²C, SPI/SD) are set to
  Hi-Z and `gpio_hold_en`'d.
- The rail-enable pin (GPIO13) is **always** the final step: driven LOW, held, and latched
  with `gpio_deep_sleep_hold_en()`.
- The rail and config pins are **never** included in the Hi-Z set (they have dedicated
  handling); input-only pins (34–39) are never treated as drive/output pins.
- The config probe (GPIO12) is decoded **active-low** by default.

## Consequences

**Positive**

- Most of the library is covered by fast, deterministic host tests with **zero hardware
  mocks** (UNBREAKABLE #2). The untested surface is reduced to the thin, review-obvious
  register-poking tail, which is instead verified by compilation for the target.
- Dual-SDK support (Arduino primary, IDF possible) falls out of using IDF APIs in the
  driver (UNBREAKABLE #1).
- The safe, maximum-savings sleep behaviour is preserved by default; power/retention
  trade-offs are opt-in and explicit at the call site.
- The authoritative, schematic-traced pin map lives in one pure header, so hardware
  revisions are a single-file change.

**Negative / trade-offs**

- A HAL-local `PowerDomainOption` enum duplicates the IDF option set and must be kept in
  sync if Espressif adds domains; the mapping table localises that risk to one file.
- The pure/driver split adds a small amount of indirection (the core emits a plan the
  driver replays) compared with inlining `gpio_*` calls directly.
- Host tests validate *decisions*, not real silicon behaviour; on-target smoke testing of
  actual `VCC_AUX` collapse remains a manual step.

## Alternatives considered

- **Mock the `gpio_*` / `esp_sleep_*` APIs and unit-test the driver directly.** Rejected:
  explicitly forbidden (UNBREAKABLE #2), and mock-based tests would assert that we called
  the functions we wrote rather than testing real behaviour.
- **Single-layer library that calls `gpio_*` inline.** Rejected: nothing would be
  host-testable without mocks, so coverage would collapse to compile-only.
- **Arduino `pinMode`/`digitalWrite` for the *whole* driver.** Rejected: there is no portable
  Arduino way to latch Hi-Z and hold state through deep sleep, so the sleep path must be IDF.
  The Arduino digital API is, however, used for plain level I/O (rail enable, config read)
  behind an `#if defined(ARDUINO)` guard with an IDF fallback — see Decision 2 — because it is
  the only reliable way to drive the RTC-capable GPIO13 pad high.
- **Bare `gpio_set_direction`/`gpio_set_level` for the rail enable (original approach).**
  Rejected after hardware testing: on the RTC-capable GPIO13 pad it did not reliably drive the
  rail high, leaving `VCC_AUX` (and the on-board LED) unpowered.
- **Hard-coded `esp_sleep_pd_config(... OFF)` with no override (status quo of
  lora-sensor).** Rejected: prevents applications that need RTC-retained memory or a wake
  stub; the configurable struct keeps the same default while removing that ceiling.
- **Expose the IDF `esp_sleep_pd_option_t` directly in the public API.** Rejected: it would
  drag `esp_sleep.h` into the host-testable header and break the native build.
